## Refuted at verification

- **Rejected close now mutates any task's state with no caller-ownership check (park + lease release)** (cas-cli/src/mcp/tools/core/task/lifecycle/close_ops.rs) — The finding's mechanics are factually accurate: on the epic branch, cas_task_close (close_ops.rs:355) calls park_task_awaiting_merge on the merge-gate Reject path with no caller-ownership check, mutating status to AwaitingMerge, clearing pending flags, and releasing the lease (line 209), where pre-change the Reject arm was a pure error return; task_claiming.rs also excludes AwaitingMerge from reclaim. However, the P1 "unauthenticated state-mutation" claim is refuted on three grounds. (1) No new attacker capability: CAS has no inter-agent trust boundary, and pre-existing tools on main already give any co-resident agent strictly stronger primitives with zero ownership checks — `task action=update status=closed` mutates any task's status, and `task action=reset force=true` force-releases any task's lease even against a live worker (main task_claiming.rs:501 explicitly comments "no ownership check") and clears the assignee; agents also have Bash access to the project-local SQLite DB. The diff adds nothing an "attacker" couldn't already do worse. (2) The park is not arbitrary damage: it fires only when factory/<assignee> genuinely has stranded commits vs the parent epic branch, so AwaitingMerge accurately describes the task's real data state; assignee is preserved, the reclaim exclusion protects the parked task from theft, and lease release is the deliberate cas-8d5b design ("park merge-gated closes, releases one-task gate") so the worker can take the next task — "stranded until supervisor merges" is the feature's intended resting state, with task action=reset as manual recovery. (3) The realistic non-assignee close caller is the supervisor, for whom parking a merge-rejected close is coherent since they are the merge authority. The residual mid-work premature-park wart requires a non-designed cross-agent close that pre-diff could already fully close or force-reset the same task, so the marginal harm attributable to this change does not sustain a P1.
