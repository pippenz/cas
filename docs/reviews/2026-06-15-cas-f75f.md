## Findings

### Data-safety paths reviewed (the high-risk hunks)

1. **Rehome guard fail-open (AC6, `check_canonical_id_rehome`, cli/cloud.rs:1162)** — ADVISORY, accepted.
   Guard reads `last_push_canonical_id` from queue metadata; refuses a push when the pinned slug changed unless `--rehome`. Fails open on metadata-read error / queue-open failure.
   - Verified the single call site (cli/cloud.rs:1408) is in the **unified `execute_push`**, which handles both personal and team pushes (team_id is an added payload field, not a separate path). So the team-project re-home scenario from the ozer bug report — the actual reported defect D — **is** covered. AC6 is complete.
   - Fail-open is defensible: it degrades to today's no-guard behavior (no regression) and the real scenario (healthy DB + operator-changed slug) reads metadata fine and is correctly refused. The bypass window (malformed DB) is not the reported threat.
   - Optional polish (not a blocker): emit a `tracing::warn!` when the guard fails open, so a malformed-DB bypass is observable.

2. **Queue drainage B/C (mark_failed/mark_synced + migrate_team_id, queue-fixer)** — OK.
   Personal path: null/invalid-payload upserts now `mark_failed` (retry_count accumulates → `failed`, stops consuming batch slots). Team path: `pending_for_team` + per-item mark_failed/mark_synced replaces the retry-count-resetting re-enqueue. `oldest_item` filters `retry_count < max_retries`. Migration normalizes legacy `team_id IS NULL` → `''` (idempotent UPDATE) so the UNIQUE constraint dedups against new `''` enqueues. Matches the live-DB finding (1,808/3,708 task rows team_id=NULL). Covered by 15 sync_queue tests.

3. **Slug resolution + ambiguity warn (AC1/2, slug-fixer)** — OK.
   `check_bucket_ambiguity` (pure, 6 unit tests) warns on stderr when resolved bucket < 10% of richest team project (≥50 memories), best-effort HTTP, silent on failure, framed advisory. `team show` now resolves via the 3-step chain (never `<not resolved>` for an active project).
