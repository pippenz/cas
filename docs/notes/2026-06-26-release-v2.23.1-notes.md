# Release v2.23.1 — PSR review lifecycle + worker isolation hardening

**Date:** 2026-06-26
**Headline:** Patch release. Closes five correctness bugs: the PendingSupervisorReview (PSR) state machine no longer loses completed work to `start`/`release`/lease races, non-isolated (standalone-task) workers can no longer commit to `main`, and the `--effort` flag is version-guarded so older `claude` CLIs don't crash.

## Why this matters
Two of these are silent data-loss class: a PSR task could be reverted to `Open` (erasing a worker's completed close) by an orphan-recovery race, and a non-isolated worker could commit straight onto a shared release branch. Both are now closed with regression tests.

## Fixes

### PSR (PendingSupervisorReview) state-machine follow-ons — EPIC `cas-a8af`
Pre-existing gaps from the `cas-b51a` Stage-1 review, all in the supervisor-review path:
- **`action=start` no longer clobbers PSR** (`cas-9684`) — starting a task that is awaiting supervisor review now returns an explanatory error instead of silently reverting it to `InProgress` and dropping it from `list status=pending_supervisor_review`.
- **Orphan-recovery exempts PSR** (`cas-6e4c`) — when a worker's lease expires and `release` fires, a PSR task is no longer reset to `Open`. The work is complete; only review is pending. (This was the silent-erase path.)
- **PSR transition releases the worker lease** (`cas-7fe9`) — the PSR early-return in the close path now releases the lease (mirroring the normal close), so a supervisor can `claim` the task for review immediately instead of waiting out a ~10-minute phantom lease.

### Worker isolation / commit guard
- **Non-isolated workers are guarded too** (`cas-ba04`) — root cause: `check_worker_git_commit_scope` early-returned `None` (allow) when `CAS_CLONE_PATH` was unset, so standalone-task workers with no worktree bypassed the guard entirely and could commit to `main`/`master`/`staging`. Branch protection now fires for **all** factory workers regardless of isolation; the cwd-outside-worktree check remains isolated-only. Complements the `cas-7e7b` denylist from v2.23.0.

### Spawn robustness
- **`--effort` flag is version-guarded** (`cas-6ee8`) — `PtyConfig::claude` now probes `claude --help` once (cached in a `OnceLock`, conservative `false` on failure) and only injects `--effort` when supported, emitting a `tracing::warn!` otherwise. Older CLIs that don't recognize the flag no longer crash silently on spawn. `CAS_FACTORY_EFFORT_SUPPORTED=1|0` overrides the probe for CI/tests.

## Verification
Assembled `main` compiles clean (10 crates); full `cargo test --no-fail-fast` = **4329 passed / 0 failed**, including the new `supervisor_review_flow.rs`, `worker_commit_guard`, and `pty` effort tests.

## Known issues
- `spawn_prep_cwd_is_worktree_for_four_workers` remains flaky under full-suite concurrency (process-global cwd race, test-only; tracked in `cas-7062`). Passed clean this release run.

## Operator note
The running factory must be rebuilt and `cas serve` restarted to pick up the new binary — the commit-guard and PSR-lifecycle fixes are daemon/MCP-side and do not take effect in an already-running session.
